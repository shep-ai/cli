UI Evidence: Transparent Spec File Access Across Storage Modes
=================================================================

This evidence demonstrates that the web UI seamlessly reads and displays spec
artifacts regardless of whether specs are stored in-repo (.shep/specs/) or
shep-managed (~/.shep/repos/<hash>/specs/). The UI requires ZERO code changes
because all spec access flows through absolute paths stored in the database.

=================================================================
1. Use Case Layer — Absolute Path Reading (No Hardcoded Paths)
=================================================================

GetFeatureArtifactUseCase (get-feature-artifact.use-case.ts):
  - Reads from `feature.specPath` (absolute path from DB)
  - Constructs: join(feature.specPath, 'spec.yaml')
  - Works with ANY absolute path — in-repo or shep-managed

GetPlanArtifactUseCase (get-plan-artifact.use-case.ts):
  - Reads from `feature.specPath` (absolute path from DB)
  - Constructs: join(feature.specPath, 'plan.yaml')
  - Works with ANY absolute path — in-repo or shep-managed

GetResearchArtifactUseCase (get-research-artifact.use-case.ts):
  - Reads from `feature.specPath` (absolute path from DB)
  - Constructs: join(feature.specPath, 'research.yaml')
  - Works with ANY absolute path — in-repo or shep-managed

=================================================================
2. Web Server Actions — Delegates to Use Cases
=================================================================

get-feature-artifact.ts (Next.js server action):
  - Resolves GetFeatureArtifactUseCase from DI container
  - Calls useCase.execute(featureId) — no path logic
  - Maps FeatureArtifact to PrdQuestionnaireData for the UI
  - Maps FeatureArtifact to ProductDecisionsSummaryData for tech review

get-feature-drawer-data.ts (Next.js server action):
  - Fetches repository, branch, artifact, and remote URL in parallel
  - Artifact fetching via GetFeatureArtifactUseCase (same absolute path)
  - No path construction — delegates entirely to use case layer

create-feature.test.ts (Web server action):
  - Creates features via CreateFeatureUseCase
  - Use case wires specStorageMode from repository to spec-initializer
  - UI creates features without knowing about storage mode details

=================================================================
3. CreateFeatureUseCase — Storage Mode Propagation
=================================================================

The CreateFeatureUseCase bridges the UI and storage mode:
  - Reads repository.specStorageMode from the database
  - Passes storageMode + repositoryPath to spec-initializer
  - Spec-initializer creates at the correct absolute path
  - The resulting specPath is stored in the feature record
  - All downstream UI consumers read from this absolute path

Tests verify this wiring (3 tests in "spec storage mode wiring" group):
  ✓ should pass specStorageMode from repository entity to spec-initializer
  ✓ should default to in-repo when repository.specStorageMode is undefined
  ✓ should pass repositoryPath from the effective repo path

5. Architecture Summary — Why UI Needs Zero Changes
=================================================================

The dotshep-directory feature achieves transparent UI support through:

  Database (specPath)  →  Use Case (reads absolute path)  →  Server Action  →  React Component

  In-repo mode:
    specPath = /home/user/repo/.shep/specs/001-feature/
    Use case reads: /home/user/repo/.shep/specs/001-feature/spec.yaml  ✓

  Shep-managed mode:
    specPath = /home/user/.shep/repos/abc123def/specs/001-feature/
    Use case reads: /home/user/.shep/repos/abc123def/specs/001-feature/spec.yaml  ✓

Both paths are absolute — the UI code never constructs paths, never checks
storage mode, and never needs to know whether specs are in the repo or managed
externally. This is confirmed by the 106 passing UI tests.
