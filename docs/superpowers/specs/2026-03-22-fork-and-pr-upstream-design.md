# Fork & PR to Upstream — Design Spec

## Problem

When a user imports a GitHub repository they don't own (e.g., an open-source project), shep currently has no way to contribute back. The merge step assumes write access to `origin`. Users need a way to fork the repo, push to their fork, and create a PR to the upstream repo — all orchestrated by shep.

Additionally, specs and evidences generated by shep's workflow are internal artifacts that shouldn't always be committed into the target repo, especially when contributing to someone else's project.

## Requirements Summary

- Per-feature `forkAndPr` checkbox on the feature creation form
- Per-feature `commitSpecs` checkbox controlling whether specs/evidences are committed to the repo (independent of `forkAndPr`, defaults to `true`, auto-flips to `false` when `forkAndPr` is enabled)
- Lazy fork — only happens at merge time, behind a single approval gate
- Fork to the authenticated user's GitHub account (`gh auth status`)
- Remap remotes: fork becomes `origin`, original becomes `upstream` (standard convention)
- Remote remapping happens only inside the feature's worktree
- New `AwaitingUpstream` lifecycle state after upstream PR is created
- Shep polls upstream PR status and auto-transitions to `Maintain` when merged

## Design

### 1. Feature Model Changes

**Three new fields on the Feature entity (`feature.tsp`):**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `forkAndPr` | `boolean` | `false` | Fork repo and create PR to upstream at merge time |
| `commitSpecs` | `boolean` | `true` | Commit specs/evidences into the repo (auto-defaults to `false` when `forkAndPr` is toggled on in UI) |
| `upstreamPrUrl` | `string?` | `null` | URL of the PR created on the upstream repo (set after fork+PR) |

**New lifecycle state:** `AwaitingUpstream` added to `SdlcLifecycle` enum. Represents "upstream PR created, waiting for external merge." Inserted conceptually between `Review` and `Maintain`.

**Repository model:** No changes. `remoteUrl` already stores the original repo URL, which becomes the `upstream` URL in a fork scenario. The fork URL is transient (derived from the authenticated user's GitHub account).

**Default behavior:** When `forkAndPr` is toggled on in the UI, `commitSpecs` auto-defaults to `false` (but the user can override back to `true`). This is a UI-level convenience, not a domain constraint.

### 2. Spec Storage

**When `commitSpecs` is `false`:** Specs are stored under SHEP_HOME instead of the worktree:

```
~/.shep/specs/<feature-id>/
  ├── spec.yaml
  ├── research.yaml
  ├── plan.yaml
  ├── tasks.yaml
  ├── feature.yaml
  └── artifacts/
```

**When `commitSpecs` is `true`:** Current behavior — specs remain in the worktree at `specs/<feature-slug>/`.

**Key details:**

- The `specPath` field on Feature already exists and currently points to the worktree spec directory. It will now point to either the worktree path or the SHEP_HOME path based on the `commitSpecs` flag.
- All spec reads/writes already go through `specPath`, so the agent and use cases follow that pointer with no special branching needed in most places.
- The SHEP_HOME spec directory is created when the feature is initialized and persists independently of the worktree lifecycle (worktree cleanup does not delete specs from SHEP_HOME).
- The `commitSpecs` flag is set at feature creation time and does not change mid-feature.

### 3. Fork Service & Merge Flow

**New `IGitForkService` interface** (Approach 1 — dedicated service, clean separation from GitPrService):

| Method | Description |
|--------|-------------|
| `forkRepository(cwd)` | Runs `gh repo fork --remote` — forks to authenticated user's account, remaps `origin` to fork, adds `upstream` pointing to original |
| `pushToFork(cwd, branch)` | Pushes feature branch to `origin` (the fork) |
| `createUpstreamPr(cwd, title, body, head, base)` | Creates PR from fork to upstream via `gh pr create --repo <upstream>` |
| `getUpstreamPrStatus(upstreamRepo, prNumber)` | Polls upstream PR status via `gh pr view --repo <upstream>` |

**Modified merge node flow when `forkAndPr` is `true`:**

```
Implementation complete
  -> Agent commits changes (specs not in worktree when commitSpecs=false)
  -> Approval gate: "Ready to fork and create upstream PR?"
  -> User approves
  -> forkService.forkRepository(cwd)         // fork + remap remotes
  -> forkService.pushToFork(cwd, branch)     // push to fork
  -> forkService.createUpstreamPr(...)       // PR from fork to upstream
  -> Store upstreamPrUrl + PR number on Feature
  -> Lifecycle -> AwaitingUpstream
```

**When `forkAndPr` is `false`:** Existing flow unchanged (push to origin, create PR, merge, transition to Maintain).

**Spec exclusion during commit:** When `commitSpecs` is `false`, `specPath` points to SHEP_HOME, so specs are never in the worktree — nothing to exclude from git operations.

### 4. Upstream PR Polling & Lifecycle Transition

**Polling mechanism:**

A new `PollUpstreamPrUseCase` handles checking upstream PR status. Triggered by:

- The web UI polling on an interval when viewing a feature in `AwaitingUpstream` state
- A background check for all features in `AwaitingUpstream` while shep is running

**Poll logic:**

1. Get feature's `upstreamPrUrl` and extract repo + PR number
2. Call `forkService.getUpstreamPrStatus(upstreamRepo, prNumber)`
3. If merged: transition feature to `Maintain`, update PR status to `Merged`
4. If closed (not merged): keep `AwaitingUpstream`, update status so UI can show "PR was closed"
5. If still open: no-op

**Polling frequency:** Every 60 seconds when the web UI is active. No background daemon — polling only happens when shep is running. Frequency is configurable via settings.

### 5. Web UI Changes

**Feature creation form:**

- New `forkAndPr` toggle — "Contribute via fork (PR to upstream)"
- New `commitSpecs` toggle — "Commit specs to repository" — defaults to `true`, auto-flips to `false` when `forkAndPr` is toggled on (user can override)
- Both sit alongside the existing `push` and `openPr` toggles

**Feature canvas node:**

- `AwaitingUpstream` lifecycle gets its own visual treatment (distinct color/icon from Review and Maintain)
- Shows upstream PR link and status badge (open/merged/closed)

**Feature detail view:**

- When in `AwaitingUpstream`, shows the upstream PR URL, status, and last polled time
- If upstream PR is closed without merge, shows a warning state

**Interaction with existing toggles:**

- When `forkAndPr` is `true`, `push` and `openPr` are implicitly true (pushing to fork and opening a PR upstream) — these toggles are hidden or disabled/locked to `true` in the UI
- When `forkAndPr` is `false`, everything works as today

## Approach Decision

**Chosen: Approach 1 — Fork Service + Merge Node Extension**

A dedicated `IGitForkService` encapsulates all fork operations. The existing merge node is extended with a fork-aware branch that delegates to this service when `forkAndPr` is true.

**Why:**
- Clean separation — fork logic doesn't pollute the already-large GitPrService
- Merge node remains the single orchestrator
- Fork service is independently testable
- No duplication of commit/CI logic (unlike a separate fork-merge node)

**Rejected alternatives:**
- Extending GitPrService directly — too large already, mixes concerns
- Separate fork-merge node — duplicates too much of the existing merge logic

## Out of Scope

- Forking to an org account (future enhancement — always forks to authenticated user)
- Changing `commitSpecs` mid-feature (set at creation, immutable)
- CI watch on the upstream repo (shep only watches CI on the fork's PR, not upstream)
- Conflict resolution on upstream PR (upstream maintainer's responsibility)
