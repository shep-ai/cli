# Fork & PR to Upstream — Design Spec

## Problem

When a user imports a GitHub repository they don't own (e.g., an open-source project), shep currently has no way to contribute back. The merge step assumes write access to `origin`. Users need a way to fork the repo, push to their fork, and create a PR to the upstream repo — all orchestrated by shep.

Additionally, specs and evidences generated by shep's workflow are internal artifacts that shouldn't always be committed into the target repo, especially when contributing to someone else's project.

## Requirements Summary

- Per-feature `forkAndPr` checkbox on the feature creation form
- Per-feature `commitSpecs` checkbox controlling whether specs/evidences are committed to the repo (independent of `forkAndPr`, defaults to `true`, auto-flips to `false` when `forkAndPr` is enabled)
- Lazy fork — only happens at merge time, behind a single approval gate
- Fork to the authenticated user's GitHub account (`gh auth status`)
- Remap remotes: fork becomes `origin`, original becomes `upstream` (standard convention)
- Remote remapping happens only inside the feature's worktree
- New `AwaitingUpstream` lifecycle state after upstream PR is created
- Shep polls upstream PR status and auto-transitions to `Maintain` when merged

## Design

### 1. Feature Model Changes (TypeSpec-first)

**TypeSpec changes in `tsp/domain/entities/feature.tsp`** — two new fields:

```typespec
/** Fork repo and create PR to upstream at merge time */
@doc("Fork repo and create PR to upstream at merge time")
forkAndPr: boolean;  // default: false

/** Commit specs/evidences into the repo */
@doc("Commit specs/evidences into the repo (defaults false when forkAndPr is enabled)")
commitSpecs: boolean;  // default: true
```

**TypeSpec changes in `tsp/domain/value-objects/pull-request.tsp`** — upstream PR data on PullRequest value object:

```typespec
/** URL of the PR created on the upstream repo (fork-and-PR flow only) */
upstreamPrUrl?: string;

/** PR number on the upstream repo */
upstreamPrNumber?: int32;

/** Status of the upstream PR */
upstreamPrStatus?: PrStatus;
```

Upstream PR data lives on the `PullRequest` value object (not on `Feature` directly) for consistency with existing PR data cohesion.

**TypeSpec changes in `tsp/common/enums/lifecycle.tsp`** — new enum member:

```typespec
/**
 * Upstream PR submitted state. The feature's code has been pushed to a fork
 * and a PR has been created to the upstream repo. Shep polls the upstream PR
 * status and auto-transitions to Maintain when merged.
 * Only reachable when forkAndPr=true.
 */
@doc("Upstream PR submitted — waiting for external merge on upstream repo")
AwaitingUpstream,
```

**Lifecycle state impact analysis:**

The new `AwaitingUpstream` state affects these existing consumers:

| Consumer | Current behavior | Required change |
|----------|-----------------|-----------------|
| Merge node (`merge.node.ts`) | Transitions `Review -> Maintain` | Add branch: when `forkAndPr=true`, transition to `AwaitingUpstream` instead |
| `PrSyncWatcherService` | Polls features in `Review` state | Extend to also poll features in `AwaitingUpstream`, calling `getUpstreamPrStatus` instead of local PR status |
| Feature agent graph | Terminates after merge node | No change — graph still terminates after merge node. `AwaitingUpstream` is a post-agent state (polling is external, not agent-driven) |
| Archive logic | Only `Maintain`, `Pending`, `Blocked` archivable | Add `AwaitingUpstream` to archivable states |
| Web UI lifecycle colors/icons | Maps each lifecycle to visual treatment | Add visual treatment for `AwaitingUpstream` |
| Feature agent state (`state.ts`) | Has `push`, `openPr` channels | Add `forkAndPr` and `commitSpecs` channels; worker populates from feature record |

**Repository model:** No changes. `remoteUrl` already stores the original repo URL, which becomes the `upstream` URL in a fork scenario. The fork URL is transient (derived from the authenticated user's GitHub account).

**Default behavior:** When `forkAndPr` is toggled on in the UI, `commitSpecs` auto-defaults to `false` (but the user can override back to `true`). This is a UI-level convenience, not a domain constraint.

**Database migration:** New migration adds `fork_and_pr BOOLEAN DEFAULT 0` and `commit_specs BOOLEAN DEFAULT 1` columns to the features table, plus `upstream_pr_url TEXT`, `upstream_pr_number INTEGER`, and `upstream_pr_status TEXT` columns to align with the PullRequest value object. Feature mapper updated accordingly.

**Immutability constraint:** `commitSpecs` is set at feature creation and cannot change mid-feature. Enforced in the update use case — reject mutations to `commitSpecs` after creation.

### 2. Spec Storage

**When `commitSpecs` is `false`:** Specs are stored under SHEP_HOME instead of the worktree:

```
~/.shep/specs/<feature-id>/
  ├── spec.yaml
  ├── research.yaml
  ├── plan.yaml
  ├── tasks.yaml
  ├── feature.yaml
  └── artifacts/
```

**When `commitSpecs` is `true`:** Current behavior — specs remain in the worktree at `specs/<feature-slug>/`.

**Key details:**

- The `specPath` field on Feature already exists and currently points to the worktree spec directory. It will now point to either the worktree path or the SHEP_HOME path based on the `commitSpecs` flag.
- All spec reads/writes already go through `specPath`, so the agent and use cases follow that pointer with no special branching needed in most places.
- The SHEP_HOME spec directory is created when the feature is initialized and persists independently of the worktree lifecycle (worktree cleanup does not delete specs from SHEP_HOME).
- The `commitSpecs` flag is set at feature creation time and does not change mid-feature.

**Spec initializer changes:** `ISpecInitializerService.initialize()` currently takes a `basePath` (worktree path) and creates `specs/NNN-SLUG/` within it. When `commitSpecs=false`, the `create-feature.use-case.ts` passes the SHEP_HOME-based path (`~/.shep/specs/<feature-id>/`) as `basePath` instead. The initializer itself needs no interface change — only the caller changes which path it provides.

### 3. Fork Service & Merge Flow

**New `IGitForkService` interface** (Approach 1 — dedicated service, clean separation from GitPrService):

| Method | Description |
|--------|-------------|
| `forkRepository(cwd)` | Runs `gh repo fork --remote` — forks to authenticated user's account, remaps `origin` to fork, adds `upstream` pointing to original. Idempotent: if already forked, `gh repo fork` returns the existing fork and adjusts remotes. |
| `pushToFork(cwd, branch)` | Pushes feature branch to `origin` (the fork) |
| `createUpstreamPr(cwd, title, body, head, base)` | Creates PR from fork to upstream via `gh pr create --repo <upstream>` |
| `getUpstreamPrStatus(upstreamRepo, prNumber)` | Polls upstream PR status via `gh pr view --repo <upstream>` |

**Fork error handling:**

| Scenario | Behavior |
|----------|----------|
| User already forked this repo | `gh repo fork` is idempotent — returns existing fork, remaps remotes. No error. |
| User cloned their own fork (origin is already a fork) | Detect via `gh repo view --json isFork,parent`. If origin is already a fork, skip forking, just ensure `upstream` remote points to parent. |
| No GitHub auth | Fail with `AUTH_FAILURE` error code (consistent with existing GitPrError pattern) |
| Fork API rate limit or network error | Fail with `FORK_FAILED` error code, include original error message |

**Modified merge node flow when `forkAndPr` is `true`:**

```
Implementation complete
  -> Agent commits changes (specs not in worktree when commitSpecs=false)
  -> Approval gate: "Ready to fork and create upstream PR?"
  -> User approves
  -> forkService.forkRepository(cwd)         // fork + remap remotes
  -> forkService.pushToFork(cwd, branch)     // push to fork
  -> forkService.createUpstreamPr(...)       // PR from fork to upstream
  -> Store upstreamPrUrl + PR number on Feature
  -> Lifecycle -> AwaitingUpstream
```

**When `forkAndPr` is `false`:** Existing flow unchanged (push to origin, create PR, merge, transition to Maintain).

**Spec exclusion during commit:** When `commitSpecs` is `false`, `specPath` points to SHEP_HOME, so specs are never in the worktree — nothing to exclude from git operations.

### 4. Upstream PR Polling & Lifecycle Transition

**Polling mechanism:**

Extend the existing `PrSyncWatcherService` (which already polls features in `Review` state at 30-second intervals) to also handle features in `AwaitingUpstream` state. When polling an `AwaitingUpstream` feature, it calls `forkService.getUpstreamPrStatus()` instead of the local PR status check. No new parallel polling service is needed.

**New `PollUpstreamPrUseCase`:**

```typescript
interface PollUpstreamPrInput {
  featureId: string;  // single feature, or
}

interface PollUpstreamPrOutput {
  status: 'open' | 'merged' | 'closed';
  transitioned: boolean;  // true if lifecycle changed
}
```

Called by `PrSyncWatcherService` for `AwaitingUpstream` features, and by the web API for on-demand status checks.

**Poll logic:**

1. Load feature, extract `pr.upstreamPrUrl` and `pr.upstreamPrNumber`
2. Call `forkService.getUpstreamPrStatus(upstreamRepo, prNumber)`
3. If merged: transition feature to `Maintain`, set `pr.upstreamPrStatus` to `Merged`
4. If closed (not merged): keep `AwaitingUpstream`, set `pr.upstreamPrStatus` to `Closed` (UI shows warning)
5. If still open: no-op

**Polling frequency:** Uses existing `PrSyncWatcherService` interval (30 seconds). Only polls when shep is running — no background daemon.

### 5. Web UI Changes

**Feature creation form:**

- New `forkAndPr` toggle — "Contribute via fork (PR to upstream)"
- New `commitSpecs` toggle — "Commit specs to repository" — defaults to `true`, auto-flips to `false` when `forkAndPr` is toggled on (user can override)
- Both sit alongside the existing `push` and `openPr` toggles

**Feature canvas node:**

- `AwaitingUpstream` lifecycle gets its own visual treatment (distinct color/icon from Review and Maintain)
- Shows upstream PR link and status badge (open/merged/closed)

**Feature detail view:**

- When in `AwaitingUpstream`, shows the upstream PR URL, status, and last polled time
- If upstream PR is closed without merge, shows a warning state

**Interaction with existing toggles:**

- When `forkAndPr` is `true`, `push` and `openPr` are implicitly true (pushing to fork and opening a PR upstream) — these toggles are hidden or disabled/locked to `true` in the UI
- When `forkAndPr` is `false`, everything works as today

## Approach Decision

**Chosen: Approach 1 — Fork Service + Merge Node Extension**

A dedicated `IGitForkService` encapsulates all fork operations. The existing merge node is extended with a fork-aware branch that delegates to this service when `forkAndPr` is true.

**Why:**
- Clean separation — fork logic doesn't pollute the already-large GitPrService
- Merge node remains the single orchestrator
- Fork service is independently testable
- No duplication of commit/CI logic (unlike a separate fork-merge node)

**Rejected alternatives:**
- Extending GitPrService directly — too large already, mixes concerns
- Separate fork-merge node — duplicates too much of the existing merge logic

## Test Scenarios

Key TDD cycles for implementation:

**Fork service (unit tests, mocked `gh` CLI):**
- `forkRepository` — happy path: forks, remaps remotes
- `forkRepository` — idempotent: already forked, returns existing fork
- `forkRepository` — origin is already a fork: detects and skips, sets upstream
- `forkRepository` — no auth: throws `AUTH_FAILURE`
- `createUpstreamPr` — creates PR to upstream repo
- `getUpstreamPrStatus` — returns open/merged/closed

**Merge node (integration tests):**
- `forkAndPr=true`: follows fork flow (commit -> approval -> fork -> push -> upstream PR -> AwaitingUpstream)
- `forkAndPr=false`: existing flow unchanged (regression)
- `forkAndPr=true` + `commitSpecs=false`: specs not in worktree, nothing to exclude

**PollUpstreamPrUseCase:**
- Upstream PR merged -> feature transitions to Maintain
- Upstream PR closed -> feature stays AwaitingUpstream, status updated
- Upstream PR open -> no-op

**Spec storage:**
- `commitSpecs=false`: specPath points to SHEP_HOME, specs created there
- `commitSpecs=true`: specPath points to worktree (existing behavior)
- `commitSpecs` immutability: update use case rejects changes

**Feature creation:**
- `forkAndPr=true` sets `push=true`, `openPr=true` implicitly
- Default `commitSpecs` value when `forkAndPr` toggled (UI-level)

**Storybook stories (mandatory):**
- Feature creation form with new toggles
- `AwaitingUpstream` canvas node visual treatment
- Upstream PR status badge (open/merged/closed states)

## Out of Scope

- Forking to an org account (future enhancement — always forks to authenticated user)
- Changing `commitSpecs` mid-feature (set at creation, immutable)
- CI watch on the upstream repo (shep only watches CI on the fork's PR, not upstream)
- Conflict resolution on upstream PR (upstream maintainer's responsibility)
